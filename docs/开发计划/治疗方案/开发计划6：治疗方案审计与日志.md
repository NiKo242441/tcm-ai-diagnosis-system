# 开发计划6：治疗方案审计与日志

## 1. 功能定位与边界

治疗方案审计与日志功能是治疗方案全生命周期管理中的独立审计模块，专注于记录、追踪和分析所有治疗方案相关操作，为医疗质量管理和合规性审查提供完整的数据追溯链。

### 1.1 功能边界声明（系统级标准 v4.0）

**本模块严格遵循**以下边界原则：
- ✅ **全生命周期审计**：记录治疗方案从计划2生成到计划7打印导出的完整操作日志
- ✅ **状态枚举统一**：所有状态必须使用计划1定义的TreatmentStatus枚举（AIGenerated=1到Archived=7）
- ✅ **版本规范审计**：记录所有版本号变更（V{主版本}.{次版本}.{修订号}）及变更原因
- ✅ **模块间调用审计**：记录计划1-7之间的所有接口调用和数据流转
- ✅ **数据完整性验证**：确保审计记录包含六大治疗体系完整数据结构
- ❌ **不处理治疗方案内容**：不修改、不生成、不编辑治疗方案内容
- ❌ **不处理状态变更**：仅记录状态变更，不主动触发状态变更
- ❌ **不处理版本号生成**：版本号由计划5统一处理，仅记录变更轨迹

### 1.2 状态流转审计（系统级标准 v4.0）

#### 1.2.1 审计状态映射表（与计划1统一状态枚举100%匹配）
| 状态枚举值 | 审计操作类型 | 数值映射 | 描述 | 来源模块 | 状态流转 |
|------------|--------------|----------|------|----------|----------|
| AIGenerated | AI_GENERATED | 1 | AI自动生成治疗方案 | 计划2 | 计划2→计划3 |
| Editing | EDIT_STARTED | 2 | 开始编辑治疗方案 | 计划3 | 计划3→计划4 |
| Checking | CHECK_STARTED | 3 | 开始配伍禁忌检查 | 计划4 | 计划4→计划5/计划3 |
| CheckFailed | CHECK_FAILED | 4 | 配伍禁忌检查失败 | 计划4 | 计划4→计划3 |
| Versioning | VERSION_CREATED | 5 | 创建新版本 | 计划5 | 计划5→计划7 |
| Finalized | FINALIZED | 6 | 标记为最终方案 | 计划5 | 计划5→计划7 |
| Archived | ARCHIVED | 7 | 归档治疗方案 | 计划5 | 计划5历史保留 |

## 2. 核心审计场景（系统级标准 v4.0）

### 2.1 治疗方案创建审计（计划2→计划3）
- **触发时机**：计划2完成AI治疗方案生成后
- **前置条件**：状态为AIGenerated(1)，版本为V1.0.0
- **审计内容**：
  - 原始症状描述（来自计划2输入）
  - AI模型版本信息（计划2统一规范）
  - 生成时间戳（CreatedAt统一字段）
  - 生成用户ID（CreatedByUserId统一字段）
  - 治疗方案版本号（V1.0.0初始版本）
- **数据格式**（符合计划1实体规范）：
```json
{
  "treatmentId": "uuid",
  "action": "CREATE",
  "source": "AI_GENERATION",
  "status": "AIGenerated",
  "statusValue": 1,
  "version": "V1.0.0",
  "createdAt": "2024-01-15T10:30:00Z",
  "createdByUserId": "ai_system",
  "details": {
    "symptoms": ["症状1", "症状2"],
    "aiModelVersion": "v2.1.0",
    "isAiOriginated": true,
    "nextModule": "计划3"
  }
}
```

### 2.2 治疗方案编辑审计（计划3→计划4）
- **触发时机**：计划3完成编辑并触发保存时
- **前置条件**：状态为Editing(2)，版本保持当前版本号
- **审计内容**：
  - 修改的字段（与计划1实体字段完全匹配）
  - 修改前后的值（完整变更轨迹）
  - 修改用户ID（CreatedByUserId统一字段）
  - 修改时间戳（CreatedAt统一字段）

### 2.3 配伍禁忌检查审计（计划4→计划5/计划3）
- **触发时机**：计划4完成配伍禁忌检查时
- **前置条件**：状态为Checking(3)，版本保持当前版本号
- **审计内容**：
  - 检查结果（通过/失败）与计划4标准完全匹配
  - 发现的禁忌项（十八反/十九畏/药性冲突等）
  - 检查时间戳（CreatedAt统一字段）
  - 检查用户ID（CreatedByUserId统一字段）
  - 检查次数计数器（TotalChecks累加）
  - 流转方向（计划5通过/计划3失败）

### 2.4 版本管理操作审计（计划5→计划7）
- **触发时机**：计划5完成版本管理并创建新版本时
- **前置条件**：状态为Versioning(5)，通过计划4检查

  - 版本创建时间戳（CreatedAt统一字段）
  - 版本次数计数器（TotalVersions累加）

### 2.5 最终方案定稿审计（计划7前置检查）
- **触发时机**：计划7准备打印导出前的最终确认
- **前置条件**：状态为Finalized(6)，版本为最终版本
- **审计内容**：
  - 最终方案完整内容快照（六大治疗体系）
  - 定稿医生身份（CreatedByUserId统一字段）

#### 2.5.1 状态枚举值强制标准（计划1统一）
所有审计记录中的状态字段必须严格按照计划1定义的枚举值：
- AIGenerated = 1（计划2输出）
- Editing = 2（计划3处理）
- Checking = 3（计划4处理）
- CheckFailed = 4（计划4输出）
- Versioning = 5（计划5输入）
- Finalized = 6（计划5输出，计划7输入）
- Archived = 7（计划5历史版本）

#### 2.5.2 版本号规范审计（计划5统一）
- 版本号格式：V{主版本}.{次版本}.{修订版本}
- 版本号变更：仅由计划5统一处理，其他模块仅记录
- 版本号验证：所有审计记录必须包含有效版本号
- 版本号关联：审计记录中的版本号必须与计划5版本记录完全一致

### 2.6 打印导出操作审计（计划7最终交付）
- **触发时机**：计划7执行打印或导出操作时

### 2.7 治疗方案删除审计
- **触发时机**：医生删除治疗方案时（物理删除或逻辑删除）
- **记录要素**：
  - 删除前治疗方案完整备份
  - 删除原因说明（必填）
  - 删除影响范围（关联的就诊记录、历史版本等）
  - 数据保留期限（符合医疗数据保存法规要求）

            CheckType = ComplianceCheckType.RegulatoryCompliance,
            EntityId = treatment.Id,
            CreatedByUserId = doctorId,
            NonComplianceItems = new List<string>
            {
                ValidateSyndromeCompleteness(treatment.SyndromePattern) ? string.Empty : "证候诊断不完整",
                ValidateTreatmentSyndromeMatch(treatment) ? string.Empty : "治疗方案与证候不匹配",
                ValidateClassicFormulaUsage(treatment) ? string.Empty : "经典方剂应用不规范"
            }.Where(s => !string.IsNullOrEmpty(s)).ToList()
        };
        
        LogComplianceAudit(auditLog);
    }
    
    public void LogHerbalPrescriptionCompliance(HerbalPrescription prescription, string doctorId)
    {
        var auditLog = new ComplianceAuditDto
        {
            CheckType = ComplianceCheckType.SafetyStandard,
            EntityId = prescription.Id,
            CreatedByUserId = doctorId,
            NonComplianceItems = new List<string>
            {
                ValidatePrescriptionFormat(prescription) ? string.Empty : "处方书写不规范",
                ValidateHerbDosageCompliance(prescription) ? string.Empty : "药材用量不合规",
                prescription.ContraindicationCheckResult?.IsSafe == true ? string.Empty : "配伍禁忌检查未通过",
                ValidateSpecialPopulationSafety(prescription) ? string.Empty : "特殊人群用药不安全"
            }.Where(s => !string.IsNullOrEmpty(s)).ToList()
        };
        
        LogComplianceAudit(auditLog);
    }

    public void LogHighRiskReviewAudit(long verificationRecordId, string reviewerId, ReviewStatus reviewStatus, string comments)
    {
        var auditLog = new ComplianceAuditDto
        {
            CheckType = ComplianceCheckType.SafetyStandard,
            EntityId = verificationRecordId,
            CreatedByUserId = reviewerId,
            Result = reviewStatus == ReviewStatus.Approved ? ComplianceResult.Passed : ComplianceResult.Failed,
            AuditComments = comments,

        };
        
        LogComplianceAudit(auditLog);
    }
    
    private bool ValidateSyndromeCompleteness(string syndromePattern)
    {
        // 检查证候诊断是否包含四诊信息
        return !string.IsNullOrEmpty(syndromePattern) && syndromePattern.Length >= 2;
    }
    
    private bool ValidateTreatmentSyndromeMatch(Treatment treatment)
    {
        // 验证治疗方案与证候的匹配性
        var syndromeKeywords = treatment.SyndromePattern?.Split('、') ?? new string[0];
        var treatmentKeywords = treatment.TreatmentPrinciple?.Split('、') ?? new string[0];
        
        return syndromeKeywords.Any(s => treatmentKeywords.Any(t => 
            IsSyndromeTreatmentMatch(s, t)));
    }
    
    private bool ValidateClassicFormulaUsage(Treatment treatment)
    {
        // 验证经典方剂使用的合理性
        if (string.IsNullOrEmpty(treatment.ClassicFormulaReferences))
            return true; // 未使用经典方剂也视为合规
            
        var formulas = treatment.ClassicFormulaReferences.Split(',');
        return formulas.All(formula => IsClassicFormulaAppropriate(formula, treatment.SyndromePattern));
    }
    
    private bool IsSyndromeTreatmentMatch(string syndrome, string treatment)
    {
        // 中医证候与治疗原则的匹配逻辑
        var matchRules = new Dictionary<string, List<string>>
        {
            ["风寒"] = new List<string> { "辛温解表", "祛风散寒" },
            ["风热"] = new List<string> { "辛凉解表", "清热解毒" },
            ["气虚"] = new List<string> { "补气", "益气" },
            ["血虚"] = new List<string> { "补血", "养血" },
            ["阴虚"] = new List<string> { "滋阴", "养阴" },
            ["阳虚"] = new List<string> { "温阳", "助阳" }
        };
        
        return matchRules.Any(rule => 
            syndrome.Contains(rule.Key) && rule.Value.Any(treatment.Contains));
    }
    
    private bool IsClassicFormulaAppropriate(string formula, string syndromePattern)
    {
        // 经典方剂与证候的适用性验证
        var formulaSyndromeMap = new Dictionary<string, List<string>>
        {
            ["桂枝汤"] = new List<string> { "太阳中风", "风寒表虚" },
            ["麻黄汤"] = new List<string> { "太阳伤寒", "风寒表实" },
            ["银翘散"] = new List<string> { "风热表证", "温病初起" },
            ["四君子汤"] = new List<string> { "脾胃气虚" },
            ["四物汤"] = new List<string> { "血虚", "血瘀" },
            ["六味地黄丸"] = new List<string> { "肾阴虚" }
        };
        
        return formulaSyndromeMap.Any(rule => 
            rule.Key.Contains(formula) && rule.Value.Any(syndromePattern.Contains));
    }
}

## 5. 数据安全与合规

### 5.1 数据完整性保障
- **防篡改机制**：所有审计日志采用数字签名，防止事后篡改
- **备份策略**：实时备份到独立存储，保留期限不少于15年
- **数据校验**：定期完整性检查，发现异常立即报警
- **访问日志**：记录所有审计日志的访问行为，防止信息泄露
- **敏感信息脱敏**：
  - 患者姓名、身份证、手机号等个人信息脱敏
  - 医生姓名脱敏（内部审计时全量，外部报告时脱敏）
  - 药品剂量、配伍等敏感医疗信息分级脱敏
  - 中医秘方、特殊疗法等知识产权保护
- **访问权限控制**：
  - 基于角色的访问控制（RBAC）
  - 基于治疗方案敏感级别的动态权限
  - 最小权限原则
  - 审计日志的审计日志（元审计）
  - 临时授权与时效性控制
- **数据加密**：
  - 传输加密（HTTPS/TLS 1.3）
  - 存储加密（AES-256数据库级加密）
  - 日志文件加密（文件系统级加密）
  - 密钥轮换机制（定期更新加密密钥）

### 5.2 权限控制体系
- **分级查看权限**：
  - 医生：查看自己操作的日志
  - 科室主任：查看本科室医生的操作日志
  - 医务科：查看全院医生的操作日志
  - 系统管理员：查看所有日志但不可修改
- **导出权限控制**：按角色限制可导出的日志范围和格式
- **匿名化处理**：用于科研或培训的日志数据需进行身份脱敏

### 5.3 合规性要求
- **医疗法规遵循**：符合《电子病历基本规范》、《医疗机构病历管理规定》
- **数据保护法规**：遵循《个人信息保护法》、《数据安全法》
- **审计标准**：符合ISO 27001信息安全管理体系要求
- **留痕要求**：满足医疗纠纷处理和法律取证的证据要求

## 6. 技术实现要点

### 6.1 技术架构
- **日志收集**：
  - 使用ASP.NET Core中间件拦截关键操作
  - 异步消息队列（RabbitMQ）处理高并发日志
  - 批量写入优化数据库性能（每100条或5秒批量提交）
  - 死信队列处理失败日志，确保数据不丢失
- **存储方案**：
  - 主数据库存储结构化日志数据（MySQL 8.0）
  - 对象存储（MinIO）存储大容量JSON快照（治疗方案完整数据）
  - 搜索引擎（Elasticsearch）支持复杂查询和全文检索
  - 冷热数据分离：近期数据MySQL，历史数据迁移至Elasticsearch
- **性能优化**：
  - 读写分离（主从数据库架构，主库写入，从库查询）
  - 分表分库（按时间维度按月分表，按科室分库）
  - 缓存热点查询结果（Redis缓存常用统计结果）
  - 数据库连接池优化（最大连接数100，超时30秒）
- **高可用保障**：
  - 主从数据库实时同步
  - 消息队列集群部署
  - 审计服务无状态设计，支持水平扩展

### 6.2 数据存储优化
- **分区存储**：按时间分区和医生ID分区，提升查询效率
- **索引策略**：针对常用查询字段建立复合索引
- **冷热分离**：近期数据保留在高性能存储，历史数据迁移到低成本存储
- **压缩存储**：对历史日志数据进行压缩，节约存储空间
