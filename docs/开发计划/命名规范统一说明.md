## 1. 命名规范概述

### 1.1 核心原则（零歧义强制版）
- **绝对一致性**：同一概念在所有层级必须使用完全相同的命名，禁止任何变体
- **语义零歧义**：每个命名必须有且仅有一个明确含义，禁止一词多义或多词同义
- **规范强制化**：100%遵循C#命名规范（PascalCase类名，camelCase字段名），无例外
- **命名简洁明确**：在保证语义完整的前提下，命名必须简洁且唯一
- **扩展预留统一**：为未来扩展预留的命名必须遵循相同规范
- **标准绝对化**：严格按照开发计划-治疗方案模块.md和输入输出.md的规范执行，禁止任何偏差

### 1.2 文档依据（强制映射规范）
- **数据字段依据**：以`输入输出.md`中明确的JSON Schema字段定义为准，必须100%精确映射
- **业务数据依据**：以`开发计划-治疗方案模块.md`中定义的数据结构和业务实体为准，禁止任何变形
- **版本控制**：所有命名必须支持版本化管理需求，版本字段必须使用标准命名
- **强制映射规则**：AI提示词字段路径与C#实体字段路径必须完全一致，禁止任何路径偏差

## 2. 实体类命名规范

### 2.1 命名规则（强制统一版）
- **实体类名**：使用PascalCase命名法，必须与业务概念完全一致，禁止缩写
- **数据库表名**：必须使用复数形式，与实体类名保持绝对一致（如Treatment→Treatments）
- **后缀规范**：禁止冗余后缀，实体名直接表达业务概念，无需添加Entity/Model等后缀
- **AI标识**：AI生成内容统一使用IsAiOriginated布尔字段标识，禁止Ai前缀
- **输入输出映射**：必须100%遵循AI提示词字段命名规范，禁止任何偏差
- **版本标识**：版本相关字段必须使用标准命名：Version、IsLatest、IsArchived、IsAiOriginated
- **命名歧义消除**：建立以下强制统一规则：
  - 治疗点位统一使用"Point"（如AcupuncturePoint、MoxibustionPoint）
  - 用量统一使用"Dosage"（遵循AI提示词映射规范，对应amount字段，禁止Quantity、Dose等变体）
  - 治疗原则统一使用"TreatmentPrinciple"（禁止简写为Principle）
  - 时间字段统一使用"CreatedAt/UpdatedAt/ArchivedAt"
  - 用户字段统一使用"CreatedByUserId/UpdatedByUserId/ArchivedByUserId"

## 4. 字段命名规范

### 4.1 通用字段强制命名（基于开发计划v3.0标准）
- **主键**：Id（int类型，自增）
- **外键**：[关联实体]Id（如：PatientId, TreatmentId, VisitId, PrescriptionId, HerbId, UserId, AcupunctureId, MoxibustionId, CuppingId, MassageId, DietaryTherapyId）
- **名称**：Name（string类型，最大长度100）
- **描述**：Description（string类型，最大长度500）
- **状态**：Status（枚举类型）
- **创建时间**：CreatedAt（DateTime类型）
- **更新时间**：UpdatedAt（DateTime类型，可空）
- **创建人**：CreatedByUserId（int类型）
- **更新人**：UpdatedByUserId（int类型，可空）
- **定稿时间**：ArchivedAt（DateTime类型，可空）
- **定稿人**：ArchivedByUserId（int类型，可空）
- **版本号**：Version（int类型，从0开始）
- **是否最新版本**：IsLatest（bool类型）

### 4.4 LLM专用解析规范（新增零歧义章节）

#### 4.4.1 实体识别算法（强制LLM解析规则）

**算法1：后缀识别法**
```
IF 实体名 ENDS WITH "Point" THEN
    归类为 "治疗点位实体"
    预期字段：PointName, Location, Coordinates
ELSE IF 实体名 ENDS WITH "Type" OR 无特定后缀 THEN
    归类为 "治疗类型实体"  
    预期字段：Method, Technique, Duration, Frequency
END IF
```

**算法2：语义验证法**
```
治疗类型实体验证：
- 必须包含：TreatmentType, Method, Technique
- 可选包含：Duration, Frequency, Dosage

治疗点位实体验证：
- 必须包含：PointName, Location
- 可选包含：Coordinates, AnatomicalPosition
```

## 6. 数据库表命名规范

### 6.1 命名规则
- **表名**：使用PascalCase命名法，复数形式（如：Patients, Treatments, Prescriptions, PrescriptionItems, AcupuncturePoints, MoxibustionPoints, CuppingPoints, DietaryTherapies）
- **字段名**：使用PascalCase命名法，单数形式（如：Name, Description, TreatmentPrinciple, PointName, Location, Method）
- **外键**：[关联实体]Id（如：PatientId, TreatmentId, PrescriptionId, HerbId, UserId, AcupuncturePointId, MoxibustionPointId, CuppingPointId）
- **索引**：IX_[表名]_[字段名]（如：IX_Treatments_PatientId, IX_PrescriptionItems_PrescriptionId_HerbId）
- **约束**：CK_[表名]_[约束描述]（如：CK_Treatments_Status, CK_PrescriptionItems_Amount）
- **联合索引**：IX_[表名]_[字段1]_[字段2]（如：IX_PrescriptionItems_PrescriptionId_HerbId, IX_Treatments_PatientId_CreatedAt）

### 6.1.1 命名冲突处理规范
- **优先级**：当字段名与C#关键字冲突时，使用后缀"Value"（如：Class → ClassValue, Object → ObjectValue）
- **缩写禁止**：禁止使用非标准缩写（如：Qty → Quantity, Desc → Description, Addr → Address）
- **语义明确**：所有命名必须具有明确语义，禁止使用模糊词汇（如：Data, Info, Item等无具体含义的词汇）
- **统一术语**：治疗点位统一使用"Point"而非"Location"或"Site"，用量统一使用"Dosage"（对应AI提示词amount字段）而非"Quantity"或"Dose"

### 6.2 LLM专用命名冲突预防策略

#### 6.2.1 强制前缀体系（LLM零歧义规范）
| 实体类别 | 强制前缀/后缀 | 命名示例 | LLM识别规则 | 冲突预防说明 |
|----------|---------------|----------|-------------|--------------|
| 治疗类型 | 无后缀或Type后缀 | `Acupuncture`, `AcupunctureType` | 识别为抽象概念 | 避免与点位混淆 |
| 治疗点位 | 强制Point后缀 | `AcupuncturePoint`, `MoxibustionPoint` | 识别为具体位置 | 零歧义标识 |

#### 6.2.2 语义层级验证算法（LLM专用）
```
一级概念（抽象层）验证：
✓ 治疗类型实体：禁止包含Point、Detail、Ingredient后缀
✓ 处方类型实体：禁止包含Point、Location后缀  
✓ 诊断类型实体：禁止包含Point、Method、Technique后缀

二级概念（具体层）验证：
✓ 治疗点位实体：必须包含Point后缀，必须包含Location字段
✓ 药材明细实体：必须包含Herb前缀，必须包含Dosage字段
✓ 诊断指标实体：必须包含Indicator后缀，必须包含Value字段

冲突检测规则：
IF 实体名 CONTAINS "Point" AND 实体名 NOT ENDS WITH "Point" THEN
    标记为命名异常，要求修正
IF 实体名 ENDS WITH "Type" AND 包含Location字段 THEN
    标记为类型错误，应改为Point实体
```

#### 6.2.3 LLM解析验证清单（强制检查项）
- [ ] **后缀验证**：治疗点位实体是否强制使用Point后缀
- [ ] **前缀验证**：药材明细是否使用Herb前缀，处方是否使用Prescription前缀
- [ ] **字段匹配**：实体字段是否与其类型语义匹配
- [ ] **层级检查**：抽象层实体是否包含具体层字段（如Location）
- [ ] **命名唯一**：是否存在同名但不同义的实体或字段

#### 6.2.4 命名冲突应急处理预案（LLM自动修复）

**当LLM识别到潜在冲突时的自动修复流程：**

**步骤1：冲突类型识别**
```
冲突类型A：Acupuncture vs AcupuncturePoint
- 检测规则：是否包含Point后缀
- 修复动作：无后缀→治疗类型，有Point后缀→治疗点位

冲突类型B：Treatment vs TreatmentPoint
- 检测规则：字段内容分析
- 修复动作：含Method字段→治疗类型，含Location字段→治疗点位

冲突类型C：Location字段误用
- 检测规则：父实体类型验证
- 修复动作：治疗类型实体中的Location字段→迁移至对应Point实体
```

**步骤2：自动重命名机制**
```
错误命名 → 自动修正命名
AcupunctureLocation → AcupuncturePoint
TreatmentPosition → TreatmentPoint  
PointMethod → Method（移除Point前缀）
TypeLocation → Point（类型修正）
```

**步骤3：验证与确认**
```
修复后必须验证：
1. 实体后缀是否符合规范
2. 字段命名是否匹配实体类型
3. JSON映射关系是否正确
4. 数据库表名是否同步更新
```
